# Homepage Specification

# Header

## Purpose

Keep the user oriented throughout a long, narrative homepage and preserve immediate access to contact and anchor navigation.

## Content

This section contains the NorthSpark Studio brand mark, links to major homepage sections, and a primary contact action. It should not introduce new messaging beyond the brand and conversion path.

## Desktop Layout

Use a sticky, centered navigation shell with the brand aligned to one side, section links centered, and the primary CTA aligned to the opposite side. The header should feel lightweight and utility-driven. It should float above the page content without covering key reading areas.

## Mobile Layout

Collapse navigation into a compact sticky header with brand, contact CTA, and menu control. The menu should open as a clear overlay or panel with large tap targets. Contact remains visible without requiring the menu.

## Motion

The header may enter softly on page load and respond to hover or tap with subtle elevation or color changes. Mobile menu open and close should be quick, readable, and reversible. Header motion must never delay access to navigation.

## Dependencies

The header supports every section. It connects the hero to deeper content and lets users jump back to Services, Process, Work, FAQ, or Contact.

# Hero

## Purpose

Establish the core value proposition, make the strategic positioning unmistakable, and offer a primary path to contact.

## Content

The hero contains the studio category, main positioning statement, a concise explanation of the integrated service, primary and secondary CTAs, and a short list of service signals. The copy role is to answer what NorthSpark does, why its approach is different, and what the visitor can do next.

## Desktop Layout

The hero should occupy the first viewport as a focused, high-confidence introduction. Content should be dominant and readable, with large display hierarchy and generous spacing. The primary CTA should sit close to the value proposition. Supporting service tags may sit beneath the CTAs as scannable proof of breadth. Background illustration should support the strategic/decision theme without competing with the headline.

## Mobile Layout

Prioritize headline, short explanation, and primary CTA. Stack actions vertically with full-width tap targets. Service tags can be reduced, hidden, or moved into a lightweight marquee only if they do not distract from the main message. The first viewport should still hint that more content exists below.

## Motion

Entrance should be the most prominent animation on the page: background settles first, then eyebrow, headline lines, supporting text, actions, and service signals. Timing should feel confident and calm. The primary CTA may have a subtle attention pulse after entrance, but it must not feel urgent or noisy.

## Dependencies

The hero starts the narrative. The next section should explain the strategic claim rather than repeat the headline.

# Decision Board

## Purpose

Turn the hero's positioning into a simple framework for how effective websites influence customer decisions.

## Content

This section contains the core decision components: message, trust, proof, and action. The content role is to make the studio's strategy tangible and memorable.

## Desktop Layout

Use a structured board or framework layout with a centered section heading and four balanced stages. Each stage should have a number, title, short explanation, and simple visual accent. The composition should read as a system, not as unrelated cards.

## Mobile Layout

Stack stages vertically in the correct reading order. Each stage should be self-contained, with separators or connectors that preserve the sense of progression. Avoid horizontal scrolling for this content.

## Motion

Reveal the section heading first, then introduce each stage sequentially. Motion should reinforce the idea of a decision path. Decorative stage shapes may appear after the text, never before it.

## Dependencies

This section depends on the hero's "reason to choose you" claim. It should prepare the user for the Problem section by defining what a successful site must handle.

# Problem

## Purpose

Identify why many websites fail to generate confidence, clarity, and action.

## Content

This section should describe common failure modes: unclear message, weak trust, generic design, and unclear CTA. The content role is diagnosis, not service explanation.

## Desktop Layout

Use a strong section introduction followed by a two-column set of problem cards or steps. Each item should include a concise title, explanation, and supporting illustration. The visual system should feel diagnostic and structured.

## Mobile Layout

Stack problem items in a single column. Keep illustrations compact so the user can scan the problem title and explanation quickly. The section should not become a long visual gallery.

## Motion

Reveal the section intro first, then bring in problem items with slight staggering. Illustrations may settle with gentle movement. Avoid playful motion that would weaken the seriousness of the diagnosis.

## Dependencies

This section follows the Decision Board by showing what happens when the decision framework is missing. It should lead naturally into Approach, where the solution philosophy begins.

# Approach

## Purpose

Explain the studio's strategic method before the user evaluates specific services.

## Content

This section should describe the way NorthSpark thinks: strategy, story/message, design/development. The content role is to show that the output is derived from customer understanding, not from a template.

## Desktop Layout

Use a linear track or staged layout that communicates progression. Each step should be visually connected but not overdesigned. Numbers and short headings should lead; explanations should support.

## Mobile Layout

Convert the track into stacked steps. Preserve the sense of order with numbering and separators. The mobile layout should keep each step readable without forcing the user to compare columns.

## Motion

Reveal steps in sequence from first to last. Connectors may draw in after the related step appears. Motion should make the method feel deliberate.

## Dependencies

Approach depends on the Problem section for context. It should lead into Services by showing that each service is selected and shaped around a goal.

# Services

## Purpose

Clarify the studio's offer categories and help users identify the type of engagement that fits their need.

## Content

This section should include service families such as landing pages, premium brand websites, Story Scrolling, and eCommerce experiences. Each service should communicate the job it performs, not only the format. Tags may clarify goal, audience, or outcome.

## Desktop Layout

Use a clear service intro followed by a grid of service cards. Cards should be equal enough to compare, with each containing a number, illustration, title, explanation, and tags. The section should include a CTA that helps unsure users ask which service fits.

## Mobile Layout

Stack service cards in a single column. Keep service titles and outcomes visible before decorative illustration detail. The CTA should become a full-width action after the cards.

## Motion

Reveal the intro before the cards. Cards may enter with small staggered timing. Hover states can add lift or emphasis, but should not make cards feel like separate landing pages. Illustrations may have restrained decorative movement only if performance remains smooth.

## Dependencies

Services depends on Approach for strategic framing. It should lead to either Story Scrolling as a featured capability or Process as the next practical question.

# Story Scrolling

## Purpose

Demonstrate how scroll can guide understanding when a message, product, or process needs to unfold in stages.

## Content

This section should explain the role of story-led scrolling and include a compact demonstration of progressive understanding. The content role is experiential differentiation.

## Desktop Layout

Use a split composition with a sticky compass or visual anchor on one side and sequential story cards on the other. The sticky area should summarize the transformation, while cards show the steps in the user's decision path. The section should feel like a guided demonstration, not a gallery.

## Mobile Layout

Remove or simplify sticky behavior. Present the compass as a normal block followed by story cards. The active card may be emphasized as it reaches the reading focus, but all content must remain accessible through standard scrolling.

## Motion

Use scroll-based focus changes to identify the current story step. The progress marker may move between stages. Cards can shift from inactive to active through opacity, scale, border, and emphasis changes. Avoid pinning on mobile unless it is proven to be smooth and accessible.

## Dependencies

This section depends on Services introducing Story Scrolling as a capability. It should lead into Process by showing that narrative experience still comes from a structured method.

# Process

## Purpose

Explain the collaboration path from discovery to launch.

## Content

This section should cover major project stages: discovery, message/structure, design/experience, development/launch. The content role is operational clarity.

## Desktop Layout

Use a four-step grid or flow with visual connectors. Each process card should include a number, short title, illustration, and explanation. The layout should communicate sequence while remaining scannable.

## Mobile Layout

Stack process cards in order and remove complex connecting lines. Cards should be compact enough that the full process does not feel intimidating. The CTA should invite a short initial discovery conversation.

## Motion

Reveal process cards in order. Connectors may draw after cards become visible on desktop. On mobile, prefer simple reveal over complex timeline animation. Hover movement should be minimal because the section is informational.

## Dependencies

Process depends on Services or Story Scrolling to create interest. It should lead into Work by setting expectations for how the displayed projects were created.

# Work

## Purpose

Prove that the studio can create credible, strategic, and visually polished websites.

## Content

This section should include selected projects with project names, short role/context descriptions, tags, links, and visual previews. The content role is proof, not exhaustive portfolio browsing.

## Desktop Layout

Use project cards with strong visual previews and supporting text. Featured projects may be larger or more visually prominent. If scroll focus is used, it should highlight the currently centered project without hiding other projects.

## Mobile Layout

Stack projects vertically. Screenshots must remain legible enough to signal real work, but copy and CTA links should be easy to reach. Avoid interactions that require hover.

## Motion

Reveal cards as they enter. Scroll focus may subtly emphasize the active project through scale, shadow, or opacity. External project links should have clear hover/focus feedback. Motion must not prevent users from opening projects quickly.

## Dependencies

Work depends on earlier sections for evaluation criteria. It should lead into About by moving from proof of work to trust in the person or studio behind it.

# About

## Purpose

Humanize the studio and reinforce the combined strategy, copy, design, and development capability.

## Content

This section should include the founder/studio context, the reason behind the studio's approach, and a compact list of strengths. The content role is trust-building and personal connection.

## Desktop Layout

Use a two-column layout: narrative copy on one side and founder/strengths elements on the other. The founder card should feel credible and restrained. Strengths should be scannable, not decorative filler.

## Mobile Layout

Stack copy, CTA, founder card, and strengths. The founder identity should appear before a long list of strengths if trust is the section's purpose. Keep the CTA accessible but avoid duplicating too many prior CTAs.

## Motion

Reveal copy and trust elements gently. Strength items may stagger in. No complex motion is needed; this section should feel stable and personal.

## Dependencies

About depends on Work for credibility and prepares the user for FAQ by making the service feel human and approachable.

# FAQ

## Purpose

Resolve common objections and practical questions before the final CTA.

## Content

This section should address timeline, scope, copywriting responsibility, design and development responsibility, mobile readiness, animation fit, Story Scrolling, redesign versus new build, pricing, and client fit. The content role is friction reduction.

## Desktop Layout

Use a section intro followed by an accordion list. Questions should be easy to scan. The contact prompt should appear after the list for unresolved questions.

## Mobile Layout

Use full-width accordion rows with large tap targets. Only one answer may be open at a time if that keeps the section manageable. Ensure expanded content does not jump unpredictably.

## Motion

Accordion open and close should be quick and readable. Icon rotation may indicate state. Answer content should be available to assistive technology and keyboard users regardless of visual animation.

## Dependencies

FAQ depends on Services, Process, Work, and About because it resolves questions those sections create. It should lead directly into Final CTA.

# Final CTA

## Purpose

Turn accumulated clarity, trust, and interest into a direct conversation.

## Content

This section should restate the core decision problem and invite a short discovery conversation. It should include a primary WhatsApp action and a secondary path back to proof if the user is not ready.

## Desktop Layout

Use a high-emphasis CTA panel with centered content, strong headline hierarchy, supporting explanation, and two clear actions. Decorative brand elements may be present but must not interfere with text.

## Mobile Layout

Use a stacked layout with right-aligned or locally appropriate reading flow. CTAs should become full-width. Decorative elements should be reduced to preserve clarity.

## Motion

Reveal the CTA panel as a confident final moment. Button hover and focus states should feel responsive. Avoid looping motion that competes with the decision.

## Dependencies

Final CTA depends on the full page journey. It should connect to Footer as a clean endpoint, not be visually swallowed by it.

# Footer

## Purpose

Provide closure, utility navigation, brand reinforcement, and one final contact path.

## Content

This section should include the logo, short brand statement, navigation links, contact action, and legal or ownership information. The content role is utility and closure.

## Desktop Layout

Use a multi-column layout with brand, navigation, and action areas. The footer should feel quieter than the Final CTA.

## Mobile Layout

Stack brand, navigation, contact, and legal text. Navigation links should become large enough for touch. The contact action should remain prominent but secondary to the Final CTA above.

## Motion

Use minimal motion. Link hover/focus states are enough. Footer should not introduce new narrative animation.

## Dependencies

Footer follows Final CTA and closes the homepage. It supports navigation recovery for users who reach the bottom.
